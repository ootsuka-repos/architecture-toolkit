# プログラミング/設計原則カタログ

各改善指摘は「どの原則に違反しているか」を必ず紐付ける（複数可）。原則名で語ることで、
指摘が普遍的根拠を持ち、レビューが教育的かつ一貫する。下の **ID** を指摘の `原則` 欄に使う。

凡例: 各原則に【定義】【違反のスメル】【主に関係する観点(viewpoints.md の番号)】【直し方】。

---

## コア4原則

### DRY — Don't Repeat Yourself
- 定義: 知識/ロジックの重複を排し、Single Source of Truth を1つに保つ。
- スメル: 同型関数の逐語コピー、定数の二重定義、コピペ分岐、「○○と同じ思想」コメント。
- 観点: 4(重複) / 6(API) / 13(設定)。
- 直し方: 共有ヘルパ/基底へ抽出し委譲（**挙動等価を確認**。差は引数で吸収）。設定は1箇所に集約。

### KISS — Keep It Simple, Stupid
- 定義: 必要最小限の複雑さに保つ。賢さより素直さ。
- スメル: 過剰な抽象/メタプログラミング、不要なデザインパターン、深いネスト、巨大な条件分岐。
- 観点: 1(凝集) / 6(抽象)。
- 直し方: 直接的な実装へ簡素化、早期 return、分岐をデータ/多態で平坦化、不要な層を削る。

### YAGNI — You Aren't Gonna Need It
- 定義: 今必要ない汎用化/拡張点/設定を作らない。
- スメル: 使われない抽象/フラグ/プラグイン機構、将来用の死にコード、過度に汎用な署名。
- 観点: 5(死にコード) / 6(抽象)。
- 直し方: 投機的な機能/抽象を削除し、必要になってから足す。死にオプション/設定を撤去。

### SoC — Separation of Concerns
- 定義: 異なる関心(I/O・ドメインロジック・表示・設定)を別モジュール/層に分離する。
- スメル: 1ファイルにロジック+I/O+表示が混在、`utils`/`common` への雑多寄せ、UI とドメインの混線。
- 観点: 1(境界) / 2(依存) / 3(命名)。
- 直し方: 関心ごとに分割、純ロジックと副作用を分離、層を切る。

## SOLID（オブジェクト/モジュール設計）

### SRP — Single Responsibility Principle
- 定義: モジュール/クラスが変わる理由は1つ。責務を1行で言えること。
- スメル: god-module、名前と中身の乖離、1関数が複数レイヤを跨ぐ。
- 観点: 1(凝集)。
- 直し方: 責務単位に分割、責務を表す名前へ。

### OCP — Open/Closed Principle
- 定義: 拡張に開き、修正に閉じる。新ケース追加で既存を書き換えない。
- スメル: 新種別ごとに巨大 if/switch を編集、コピペで拡張するしかない硬直。
- 観点: 6(抽象) / 1。
- 直し方: 多態/戦略/テーブル駆動で拡張点を作る（ただし YAGNI とのバランス）。

### LSP — Liskov Substitution Principle
- 定義: 派生は基底の契約を破らず置換可能であること。
- スメル: サブクラスが例外を投げる/契約を狭める、isinstance 分岐の氾濫。
- 観点: 6。
- 直し方: 契約を揃える、継承より合成、インターフェースを正しく設計。

### ISP — Interface Segregation Principle
- 定義: 利用者に不要なメソッドを強制しない。小さく的を絞ったインターフェース。
- スメル: 巨大インターフェース、使われないメソッドの実装義務、神オブジェクト。
- 観点: 6。
- 直し方: 役割別に分割。

### DIP — Dependency Inversion Principle
- 定義: 上位/下位とも抽象に依存。下位の具体に上位が縛られない。依存方向を制御する。
- スメル: レイヤ違反(基盤が上位を import)、具体への直接依存、循環依存。
- 観点: 2(依存)。
- 直し方: 抽象(インターフェース/引数/イベント)を挟む、共有を下位へ抽出、依存を逆転。

## 結合・凝集・カプセル化

### High Cohesion / Low Coupling
- 定義: モジュール内は強く関連し、モジュール間は弱く結合。
- スメル: 無関係要素の同居(低凝集)、過度な相互 import(高結合)、変更が波及する。
- 観点: 1 / 2。
- 直し方: 関連を集約、境界を疎結合化(引数/イベント/インターフェース)。

### Information Hiding / Encapsulation
- 定義: 実装詳細を隠し、最小の公開面のみ晒す。
- スメル: 内部状態の直接露出、巨大な公開API、private 詳細への外部依存。
- 観点: 6。
- 直し方: 公開面を絞る、アクセサ/境界で内部を隠す。

### Law of Demeter（最小知識）
- 定義: 直接の隣人とだけ話す(`a.b.c.d` を避ける)。
- スメル: 深いプロパティチェーン、他オブジェクトの内部構造への依存。
- 観点: 2 / 6。
- 直し方: Tell-Don't-Ask、必要な操作を隣人に委譲。

### Composition over Inheritance
- 定義: 振る舞いの再利用は継承より合成で。
- スメル: 深い継承木、再利用目的だけの継承、フラジャイル基底クラス。
- 観点: 6。
- 直し方: 機能を部品として注入/合成。

### Orthogonality
- 定義: 独立した関心は独立に変更できる(一方の変更が他に波及しない)。
- スメル: 変更の連鎖、隠れた結合、グローバル状態の共有。
- 観点: 1 / 2 / 10。
- 直し方: 状態を局所化、依存を明示、副作用を隔離。

## 振る舞い・堅牢性

### Single Source of Truth (SSOT)
- 定義: 各事実/設定の正本は1つ。
- スメル: 同じ定数/設定/規約が複数箇所、コピーされた一覧。
- 観点: 4 / 13.
- 直し方: 正本を1つに、他は参照/導出。

### Principle of Least Astonishment（驚き最小）
- 定義: 名前/挙動は利用者の期待どおりであること。
- スメル: 名前と中身の乖離(`sampler` が実はサムネ)、副作用のある getter、同名異義ファイル。
- 観点: 3(命名) / 14.
- 直し方: 用途を表す名へ改名、副作用を明示/分離。

### Fail Fast / Fail Loud
- 定義: 不正は早く・明確に失敗させ、握り潰さない。
- スメル: `except: pass`、生例外の素通し、原因を隠すフォールバック、サイレント無効化。
- 観点: 7(エラー処理)。
- 直し方: 早期検証、意味あるエラーへ整形、回復不能は即 fail、フォールバックは同種失敗に限定。

### Tell-Don't-Ask / Command-Query Separation (CQS)
- 定義: 状態を取り出して判断するより振る舞いを委ねる。問い合わせと変更を分離。
- スメル: 取得→外部で判断→書き戻し、副作用のあるクエリ。
- 観点: 2 / 6.
- 直し方: 操作をオブジェクトに委譲、query と command を分ける。

### Robustness / Postel's Law（控えめに送り寛容に受ける）
- 定義: 入力に寛容、出力は厳格。境界で正規化。
- スメル: 入力前提が暗黙、境界での未検証。
- 観点: 7 / 11.
- 直し方: 境界で検証/正規化、内部は厳格な型で扱う。

### Idempotency（冪等性）
- 定義: 同じ操作を複数回適用しても結果が同じ。
- スメル: 再実行で二重生成/破損、リトライ安全でない。
- 観点: 10.
- 直し方: 操作を冪等に設計(存在チェック/upsert/トランザクション)。

## 設定・運用・進化

### Convention over Configuration
- 定義: 妥当な規約を既定にし、設定を最小化。
- スメル: 過剰な設定項目、毎回必要なボイラープレート。
- 観点: 6 / 13.
- 直し方: 既定を賢く、必要時のみ上書き可能に。

### Least Privilege / Defense in Depth
- 定義: 最小権限。多層防御。
- スメル: 過剰権限、単一防御線、秘密の平文/ログ露出。
- 観点: 11(セキュリティ)。
- 直し方: 権限を絞る、入力検証を多層化、秘密を隔離/マスク。

### Boy Scout Rule（来たときより綺麗に）
- 定義: 触れた箇所を少し改善して去る。
- スメル: 化石コメント/旧名/デッドの放置、TODO の堆積。
- 観点: 5 / 14.
- 直し方: 触れたついでに死にコード/旧名/誤誘導コメントを除去。

### Testability / Design for Test
- 定義: テスト可能に設計する(seam・純関数・依存注入)。
- スメル: I/O 直結で単体化不能、グローバル依存、巨大関数。
- 観点: 8(テスト)。
- 直し方: 純ロジック抽出、依存注入、特性化テスト先行。

## 使い方
- 各指摘に最も該当する原則 **ID** を1〜3個付ける（例: `DRY, SSOT` / `SRP, SoC` / `Least Astonishment`）。
- 原則は観点(viewpoints.md)と直交する補助軸。観点=「どこを見るか」、原則=「なぜ問題か(普遍的根拠)」。
- 1つの根本原因が複数原則に跨ることは多い(例: god-module = SRP + SoC + High Cohesion 違反)。束ねて根本対処を推奨。
- 原則名は**断罪のためでなく説明のため**に使う。なぜ直すべきかが利用者に伝わることが目的。
