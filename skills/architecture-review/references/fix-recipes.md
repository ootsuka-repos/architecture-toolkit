# 修正レシピ集（AI が実行しやすい再現手順）

各指摘の「直し方」は、別の AI エージェント（または自分）がそのまま着手できる粒度で書く。
変更タイプ別に**前提・手順・波及更新・検証**を定型化したのが本レシピ。指摘の `直し方` 欄からは
該当レシピを参照しつつ、その指摘固有の `file:line`・before→after を埋める。

共通ルール:
- **正確な場所**: ファイルパス＋（行範囲 or 一意に特定できるアンカー文字列）。行は変動するのでアンカー併記を推奨。
- **before → after**: 変更後の姿を最小スニペットで示す（曖昧な散文より差分が伝わる）。
- **波及(refs)**: その変更で更新が要る他箇所（import / エントリ定義 / 設定 / ドキュメント / 動的参照）を列挙。
- **順序(deps)**: 先に済ませるべき変更があれば明記（特にリネームは全域に波及）。
- **検証**: 変更後に走らせる具体コマンド（成功条件つき）。

---

## R1. リネーム / 移動（命名・構成改善）
- 前提: import がリポジトリ全域に波及。並列だと共有 importer で衝突 → **中央で一括・段階検証**。
- 手順:
  1. 対象内の相対 import を絶対化（`from .X`→`from pkg.sub.X`）して、移動を文字列置換で済むようにする。
  2. `git mv <old> <new>`。
  3. 絶対 dotted パス `pkg.old` → `pkg.new` を .py / 設定(pyproject等) / ドキュメントで全置換。
  4. 必要なら呼び出し側を `from pkg.new import name as old_alias` にして属性参照を非改変に。
- 波及: import / pyproject `[project.scripts]` / package-data / lint exclude / `.gitignore` /
  README / docstring / **動的参照**(`importlib`/`__getattr__`/`python -m ...` 文字列)。
- 落とし穴: `from pkg import name`(パッケージ経由形)は dotted 置換で漏れる→実 import で発覚。移動後にスモーク必須。
- 検証: import 解決チェック → ビルド/型 → リンタ → 実 import スモーク → 旧名 grep が0件。

## R2. 重複の共通化（DRY/SSOT）
- 前提: **挙動が厳密に等価**であること（例外文言・副作用・順序・既定値の差に注意）。
- 手順:
  1. 重複箇所を全て特定（grep）。差分を洗い出す。
  2. 共有ヘルパ/基底へ抽出。差は引数で吸収（無理に1関数化して挙動を変えない）。
  3. 各箇所を委譲に置換。重複定義/再エクスポートを削除。
- 波及: 旧シンボルの import、再エクスポート、テスト。
- 等価性確認: テストがあれば実行。無ければ入出力を突き合わせる/特性化テストを先に書く。差があれば共通化しないか引数化。
- 検証: リンタ(未使用 import) → 実 import スモーク → （あれば）テスト。

## R3. 死にコード削除（YAGNI/Boy Scout）
- 前提: **本当に未参照**を機械確認（import 到達可能性グラフ＋動的参照 grep）。手動CLI/デバッグツールは機能なので残す。
- 手順: 未参照ファイル/関数/エンドポイント/設定を削除。化石コメント・旧名 docstring も除去。
  未配線だが価値ある機能は「削除」か「配線して機能化」を判断して明記。
- 波及: エントリ定義・設定・ドキュメントから当該記述も削除。
- 検証: import 解決 → リンタ → 旧名 grep 0件。

## R4. god-module 分割（SRP/SoC/凝集）
- 前提: 「意図的な薄い分割」を誤認しない（docstring/履歴で意図確認）。
- 手順:
  1. ファイル内の責務を列挙（純ロジック / I/O / 表示 / 設定 …）。
  2. 関心ごとに新モジュール/サブパッケージへ分離（純関数を抽出して seam を作る＝テスト性も上がる）。
  3. 公開面を `__init__` 等で整理。
- 波及: import 更新（R1 と同型）、テスト。
- 検証: import 解決 → リンタ → スモーク。

## R5. 依存方向の是正（DIP/Low Coupling）
- 前提: 循環/レイヤ違反を import グラフで特定。
- 手順: 共有を下位へ抽出 / 抽象(インターフェース・引数・イベント)を挟む / 依存を逆転。下位が上位を import しない形へ。
- 波及: 双方の import、抽象の置き場（基盤層）。
- 検証: import グラフで循環/逆流が消えたことを確認。

## R6. エラー処理の堅牢化（Fail Fast/Postel）
- 手順: `except: pass`/生例外素通しを、(a) 早期検証 (b) 意味あるメッセージへ整形 (c) 回復不能は即 fail
  (d) フォールバックは同種失敗に限定、へ。接続不可/timeout も整形。原因をログに残す。
- 場所の示し方: `try`/`except` の行範囲を明示し、before(握り潰し)→after(整形+再送出/分岐)を示す。
- 検証: 該当経路の異常系を再現できれば実行、無ければレビューで挙動説明。

## R7. 設定集約（SSOT/Convention over Configuration）
- 手順: 散在する定数/env 読み取りを1モジュール(`const`/設定)へ集約。賢い既定＋必要時上書き。未使用 env を削除。
- 波及: 各参照を集約先へ、`.env`/ドキュメント。
- 検証: 参照の grep が集約先のみ、未使用 env が消えたこと。

## R8. テスト導入（Testability）— 挙動変更系の前提
- 手順: 変更前に**特性化テスト**(現在の入出力を固定)を書く → リファクタ → テストが緑のままを確認。
  単体化できない箇所は純関数を抽出して seam を作ってからテスト。
- 検証: テスト実行（リファクタ前後で同結果）。

---

## 大規模・多数の修正をまとめて回すとき
- リネーム/移動の一括は **1グループずつ→各段で検証**（壊れた状態で次に進まない）。
- 挙動を変える整理（R2/R4/R6）は、ファイルが**重複しないグループ**に分けて並列実行も可。共有 importer・
  設定・エントリは中央で更新して衝突を避ける。
- どの場合も最後に: import 解決 / ビルド・型 / リンタ / 実 import スモーク / 旧名 grep の全 PASS を確認。
