# 観点別チェックリスト（網羅監査の中核）

Phase 2 で各観点をひと通り当てるための問い・スメル・検出法・直し方。
全観点を機械的に一巡することで「網羅性」を担保する。指摘は必ず根拠(`file:line`)・深刻度・労力付きで。

---

## 1. モジュール境界と凝集 (cohesion / boundaries)
- 問い: 各パッケージ/モジュールの責務を1行で言えるか。1ファイルが複数の無関係な関心を抱えていないか。
- スメル: god-module(数百行・多目的)、`utils`/`common`/`helpers` への雑多な寄せ集め、名前と中身の乖離、
  「設定」と言いつつロジック、1関数が複数レイヤを跨ぐ。
- 検出: 大きいファイルを行数で抽出し責務を要約させる。`common.py` 等の中身が単一テーマか確認。
- 直し方: 関心ごとに分割 / 責務を表す名前へ / 純ロジックと I/O を分離 / 適切なサブパッケージへ移動。
- 注意: **意図的な薄い分割**(役割別ファイル)を god-module と誤認しない。docstring/履歴で意図を確認。

## 2. 結合と依存 (coupling / dependency direction)
- 問い: 依存方向は一方向か(基盤→ドメイン→アプリ→UI)。循環依存は無いか。下位が上位を import していないか。
- スメル: レイヤ違反(core が web を import)、循環 import、`from . import x` 経由の密結合、
  漏れた抽象(下位が上位の型/概念を知っている)、過度な相互 import。
- 検出: import グラフを作る(`methodology.md`)。循環検出。レイヤ跨ぎの逆流を探す。
- 直し方: 依存を逆転(DIP)/共有を下位へ抽出/インターフェース分離/イベントや引数で疎結合化。

## 3. 命名と発見可能性 (naming / discoverability)
- 問い: ファイル/フォルダ/関数/変数名が「何で・どう使うか」を表すか。命名規則は一貫しているか。
- スメル: 同名ファイルが複数(`step.py`×4, `parse.py`×2)、用途と乖離(`sampler` が実はサムネ生成)、
  化石接頭辞(廃止コマンド名)、`_v2`/`_new`/`_old`、和製英語の誤読(`name`=ネーム)、CLI名と実装パスの不一致。
- 検出: ファイル名一覧を俯瞰、同名/紛らわしい名の grep、docstring の旧名残存。
- 直し方: 役割を表す名へ改名(import/エントリ/動的参照/docstring も全更新)。サブパッケージで名前空間を分離。
- 注意: 改名は import がリポジトリ全域に波及。**中央で一括・段階検証**(`methodology.md` の手順)。

## 4. 重複と共通化 (DRY / shared abstractions)
- 問い: 同型ロジックが複数箇所に逐語コピーされていないか。共通化が途中で止まっていないか。
- スメル: ほぼ同一の関数/比較ブロック、HTTPセッション/Cookie/JSON抽出/正規化の各自実装、
  「○○と同じ思想」コメント(共有すべき兆候)、定数の二重定義。
- 検出: 似た関数名/シグネチャ/定数を grep、コメント参照を追う。
- 直し方: 共有ヘルパ/基底へ抽出し委譲。**ただし挙動が厳密に等価か確認**(例外文言・副作用・順序の差に注意)。
  テストが無いなら等価性をコードで確認、または特性化テスト先行。差があるなら引数で吸収。

## 5. 死にコード/陳腐化 (dead code / vestigial)
- 問い: どこからも呼ばれないファイル/関数は。旧版の残骸・到達不能エンドポイント・化石コメントは。
- スメル: import グラフ上の孤立、`if False`、未配線のAPI、廃止機能のenv/設定、TODO化石。
- 検出: **import 到達可能性グラフ**(エントリから到達不能を列挙, `methodology.md`)、
  シンボル単位は grep で被参照0を確認。動的参照(importlib/subprocess文字列/エントリ定義)も忘れず確認。
- 直し方: 削除。ただし「手動CLI/デバッグツール(import されないが python -m で起動)」は機能なので残す。
  未配線でも価値ある機能は「配線して機能化」も選択肢。判断を明記。

## 6. 抽象とAPI設計 (abstraction / public surface)
- 問い: 公開API/インターフェースは最小十分か。設定値は1箇所に集約されているか。拡張点は明確か。
- スメル: 巨大な公開面、設定のハードコード散在、神オブジェクト、不要な汎用化(YAGNI)、
  逆に「コピペで拡張」しか道がない硬直。
- 検出: `__init__`/エクスポート、設定読み取り箇所(env/const)の散らばりを確認。
- 直し方: 設定を集約(`const`/設定モジュール)、公開面を絞る、拡張は合成/戦略で。過剰抽象は削る。

## 7. エラー処理と回復性 (error handling / resilience)
- 問い: 失敗モードは把握されているか。例外は適切に伝播/整形されるか。握り潰しは無いか。フォールバックは妥当か。
- スメル: `except Exception: pass`、生の例外をそのままユーザーへ、原因を隠すフォールバック、
  リトライ無し/無限リトライ、部分失敗の未処理、タイムアウト/接続不可の未整形。
- 検出: `except`/`try` を grep、外部I/O・サブプロセス・ネットワーク呼び出しの周辺を確認。
- 直し方: 失敗を分類し意味あるメッセージに整形、回復不能は早期 fail、フォールバックは「同種の失敗」に限定、
  リトライは上限と区別付き。ログに原因を残す。

## 8. テストと検証可能性 (testing / testability)
- 問い: テストは存在するか/何を守るか。挙動を変える変更を安全に行える seam があるか。
- スメル: テスト皆無、I/Oやモデルに直結で単体化不能、巨大関数、グローバル状態依存。
- 検出: `tests/` の有無・カバレッジ・テスト実行手段。純ロジックと副作用の分離度。
- 直し方: まず**特性化テスト**(現挙動を固定)→ リファクタ。純関数を抽出して seam を作る。CI で最低限の検証。
- 注意: テスト不在のコードベースでは「挙動を変える改善」は高リスク。レポートに前提として明記。

## 9. 性能と資源 (performance / resources)
- 問い: ホットパスは。重い初期化(モデル/接続)が毎回走っていないか。I/O・並行度・メモリ/VRAM は妥当か。
- スメル: ジョブ毎のモデル再ロード、N+1 I/O、不要な再計算、同期I/Oでブロック、二重ロードでメモリ枯渇、
  逐次で並列化余地。
- 検出: 初期化/ロード箇所、ループ内I/O、プロセス境界でのロード重複を追う。実測できるなら計測。
- 直し方: 重い資源は常駐化/キャッシュ/単一コピー化、バッチ化、非同期化、適切な並行度。量子化等で資源削減。

## 10. 並行性と状態 (concurrency / state)
- 問い: 共有可変状態は。競合/データ競合は。プロセス/スレッド境界は明確か。冪等性は。
- スメル: グローバル可変、ロック無しの共有書き込み、プロセス間で前提を共有、再入で壊れる、
  サブプロセスとin-processの混在で状態不一致。
- 検出: モジュールグローバル、`threading`/`multiprocessing`/`asyncio`、共有キャッシュ周辺を確認。
- 直し方: 状態を局所化/不変化、ロックや単一所有者(常駐ワーカー)、境界を明示、冪等に設計。

## 11. セキュリティと秘匿 (security / secrets)
- 問い: 外部入力は検証されるか。秘密情報(キー/Cookie)は安全に扱われ、誤ってコミット/ログされないか。認可は。
- スメル: パストラバーサル未対策、秘密の平文コミット/ログ出力、未検証の subprocess 引数/URL、過剰権限。
- 検出: ファイル配信・パス結合・subprocess・外部API・`.env`/秘密の扱いを確認。`git` 追跡の秘密ファイル。
- 直し方: 入力検証/正規化、秘密は env/secret store・gitignore、ログでマスク、最小権限。
- 注意: セッション中に秘密(キー等)を出力しない。コミット/push 前に秘密混入を確認。

## 12. 可観測性 (observability)
- 問い: 障害時に原因を切り分けられるか。ログ/トレース/進捗は十分か(過多でないか)。
- スメル: ログ皆無 or ノイズ過多、進捗が「running」だけ、エラーに文脈なし、相関ID無し。
- 検出: ロギング箇所、ジョブ進捗の粒度、エラー時の情報量を確認。
- 直し方: 構造化ログ、重要遷移にログ、進捗の粒度を上げる、エラーに入力/段階を添える。

## 13. ビルド/パッケージング/依存/設定 (build / packaging / deps / config)
- 問い: 依存定義は正確か(過不足)。エントリ/パッケージ定義は実態と一致か。再現性はあるか。env は集約され文書化済みか。
- スメル: エントリが存在しないモジュールを指す、package-data/exclude の古いパス、未使用依存、
  未使用/廃止 env、ベンダーをツール(lint/test/build)が誤って処理。
- 検出: pyproject/package.json の scripts・packages・data・依存、`.gitignore`、lint/build の除外設定。
- 直し方: エントリ/glob/exclude を実態へ、未使用依存・env を削除、ベンダー/生成物を除外、env を1箇所に集約し文書化。

## 14. ドキュメントとオンボーディング (docs / onboarding)
- 問い: README/設計図は実態と一致するか。新規参画者が主要フローを追えるか。命名・構成が自己説明的か。
- スメル: 旧構成のままのREADME、存在しないコマンド、docstring の旧名、入口が不明。
- 検出: README とディレクトリ実態の差分、docstring の旧称 grep。
- 直し方: 構成図/フロー/エントリを実態に更新、各パッケージの責務を1行で、命名で自己説明。

---

## 観点の使い方
- 全観点を一巡し「該当なし」も含め判断する(網羅)。該当ありは指摘化、無しは触れなくてよい。
- 1指摘=1観点に割り当て、深刻度×労力で優先度化。複数観点に跨る根本原因(例: god-module が結合・テスト性・命名を悪化)は束ねて根本対処を推奨。
