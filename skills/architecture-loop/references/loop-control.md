# ループ制御（収束条件・予算・進捗・停止判定・台帳）

本スキルの心臓部。「いつ止めるか」を**着手前に契約として固定**し、毎周末に機械的に判定する。
停止条件のない自動ループは暴走するので、**ゲート・予算・頭打ちの三重で必ず有限回で止まる**ようにする。

## 1. 停止条件（優先順に評価）
毎周末、次を上から評価し、最初に該当したもので停止する:

| # | 条件 | 種別 | 意味 / 次アクション |
|---|------|------|----------------------|
| 1 | **回帰**: 周末の全体検証が赤、wave 切り戻し後も赤 | 失敗 | 即停止。壊れた変更を切り分けて報告。緑だった周までを成果として残す。 |
| 2 | **収束**: 品質ゲート達成(例 critical=0 かつ high=0) | 成功 | ゴール到達。最終レポートへ。 |
| 3 | **頭打ち**: 今周の解消(critical+high)が進捗閾値未満 | 成功(逓減) | 費用対効果が低いので停止。残課題を later として報告。 |
| 4 | **振動**: 同一 finding が出戻る/純増のないファイル往復 | 失敗 | ループでは解けない。停止して手動対応に回す。 |
| 5 | **予算消尽**: 反復予算(最大イテレーション)に到達 | 打ち切り | 安全弁。残課題と次の一手を報告。 |

回帰と振動は**失敗系**(構造的に止める)、収束と頭打ちは**成功系**(目的達成 or 逓減で止める)、
予算消尽は**打ち切り**(上限による強制停止)。レポートにはどれで止まったかを必ず明記する。

## 2. 既定パラメータ（Phase 0 で確定・上書き可）
- `quality_gate`: `critical == 0 && high == 0`（収束の定義。観点限定なら「対象観点の指摘 ≤ N」等に置換)。
- `max_iterations`(反復予算): **5**（安全弁。必ず有限）。
- `progress_threshold`(進捗閾値): **1**（1周で解消した critical+high がこれ未満なら頭打ち)。
- `apply_policy`(各周の対象): `now`+`next` の `severity ∈ {critical, high}`。`later`/medium 以下は既定で対象外。
- `full_sweep_every`: **3**（差分スコープで回しつつ、3周に1回は全体を再スイープし取りこぼしを防ぐ)。

これらは「契約」。ユーザー指定があれば上書きし、レポート冒頭に実際に使った値を明記する。

## 3. 進捗(デルタ)指標
各周で review JSON から severity 別件数を集計し、前周との差を測る。
- `counts(n) = {critical, high, medium, low}`（その周の review 時点の件数)。
- `resolved(n) = max(0, (critical+high)(n-1) − (critical+high)(n))`（高優先の純減)。
- `progress(n) = resolved(n)`。`progress(n) < progress_threshold` なら頭打ち(停止条件 #3)。
- 併せて `applied / failed / skipped` の件数(その周の fix 実績)も記録する。**review 上の件数(残り)** と
  **fix 実績(その周直した数)** は別物。収束判定は前者(残りが基準以下か)で行う。
- principle_coverage の violated 数も推移として残す(構造的負債が減っているかの別軸)。

注: 件数は「同じ finding を ID で対応付けて」差分を取るのが理想。難しければ severity 別総数の純減で近似する。

## 4. 台帳（周回間で持ち越す状態）
findings を ID で追跡する台帳を持ち、周をまたいで状態を更新する。`arch-loop/ledger.json` に永続化:

```json
{
  "F01": { "status": "applied",  "iter": 1, "commit": "abc123" },
  "F02": { "status": "failed",   "iter": 1, "attempts": 1, "reason": "before/after が実コードと乖離" },
  "F07": { "status": "deferred", "reason": "later。今回のポリシ対象外" },
  "F09": { "status": "wontfix",  "reason": "意図的な設計と確認。再掲しない" }
}
```
- `applied`: 適用済み・検証緑。次周の review では原則再掲されないはず(出たら振動の兆候)。
- `failed`: 適用したが検証赤/乖離大。**`attempts` を数え、上限(既定 2)に達したら `wontfix` に昇格**して
  再挑戦しない(振動防止)。
- `deferred`: ポリシ対象外(later/低優先)。予算が残れば最終周以降で拾うことを検討。
- `wontfix`: 直さないと決めたもの(意図的設計・再挑戦上限到達)。次周 review に「再掲不要」として渡す。

## 5. 振動の検知（停止条件 #4 の判定）
ループが「進んでいるつもりで進んでいない」状態を検出する:
- **出戻り**: ある周で `applied` にした finding が、後の周の review で再び(同一 ID か同一場所・同一指摘で)
  critical/high として出る → 適用が表層的か、別の修正と打ち消し合っている。
- **純増ゼロの往復**: 複数周で同じファイル集合を編集しているのに `progress` が 0 が続く(既定 2 周連続)。
  ただし `progress_threshold ≥ 1`(既定 1)のときは progress==0 の最初の周で #3(頭打ち)が先に停止するため、この『2周連続』判定が意味を持つのは進捗 0 周も継続を許す `progress_threshold = 0` 設定のときに限る。閾値≥1 運用では本サブ条件ではなく上記『出戻り』で振動を捉える。
- **failed の堂々巡り**: 同じ finding が `attempts` 上限まで失敗を繰り返す。

いずれも検出したら停止し、該当 finding を「ループでは解けない/人手判断が要る」として報告する。

## 6. 回帰の扱い（停止条件 #1）
- 周末(または周内 wave 後)の全体検証が赤 → まずその wave/周の変更を `git restore`/`git reset` で切り戻す。
- 切り戻して緑に戻るなら、その finding を `failed` にして**その周は緑の範囲のみコミット**、原因を報告して停止。
- 切り戻しても赤(ベースラインが既に壊れていた等)なら、ループの問題ではないので即停止し切り分けを促す。
- **壊れた状態のコミットは作らない/次周に進めない**。緑の周までを確定成果として残す。

## 7. 自己点検（Phase 0 終了時）
- 品質ゲート・反復予算・進捗閾値・適用ポリシがすべて具体値で決まっている(暗黙のデフォルトに頼らない)。
- 予算は有限(`max_iterations` が設定済み)で、停止条件が最低でも「ゲート/頭打ち/予算」の3系統揃っている。
- 成果物の置き場(`arch-loop/iter-<n>/`・`ledger.json`)が決まっている。
- 適用前ゲート(作業ツリー clean・ブランチ・ベースライン緑)を通している。
