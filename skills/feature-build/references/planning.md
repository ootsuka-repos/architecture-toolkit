# planning.md — 実装タスクの wave 分割（Phase 2）

設計(design.md のアウトプット)を「並列実行できる実装タスクの列」に落とす。基本発想は architecture-fix の
wave 分割と同じ: **同一ファイルに同時に触れない単位に分け、依存順を守り、貪欲に詰める**。

## task の定義
各実装 task は次を持つ:
- `id` — 安定 ID(`T1`,`T2`,…)。
- `title` — 何を作るか(1行)。
- `spec` — 設計から渡す契約(型/シグネチャ/エラー型)とやること。
- `files[]` — **触れるファイル集合**(新規作成 + 既存編集)。wave 分割の衝突判定の核。
- `deps[]` — 先に完了が要る task の id(基盤・契約を提供する task)。
- `tests` — この task で書く/通すテスト。
- `verify` — 完了判定コマンドと成功条件(verify.md 参照)。

## ファイル集合の解決
`files[]` = 新規作成するファイル ∪ design.md の影響範囲で挙げた既存編集ファイル。
- できるだけ**細かく**割って重なりを減らすと並列度が上がる。
- どうしても複数 task が同じ既存ファイル(例: ルータ定義)に触れるなら、その編集は**配線 task に集約**して切り出す。

## 分割の原則
1. **基盤先行(wave0)**: 他 task が import する型/スキーマ/インターフェース/共通ユーティリティを最初の wave に置く。
   ここで契約が確定するので、後続 task はそれを所与に並列実装できる。
2. **機能本体(中間 wave)**: 契約に依存する各実装(ハンドラ/サービス/画面部品/ロジック)を、
   **ファイルが重ならない task を貪欲に同一 wave へ詰めて**並列化する。
3. **配線は最終 wave で中央集約(SERIAL)**: routing 登録・DI/コンテナ登録・export/index 追記・設定・エントリ接続は
   同じファイルに集中しがちなので、最後に中央で1つずつ繋ぎ込む。並列化しない。
4. 結果は `[wave0, wave1, …, wireWave]`。wave 内は並列、wave 間は逐次(前段検証 PASS 後に次段)。

## テスト計画
- **テスト同伴が既定**: 各 task は実装と一緒にテストを書く(受け入れ条件→テストケースへ写像)。
- **重要パスはテスト先行(TDD)**: 失敗するテストを先に書き、実装で緑にする。設計の契約が安定している基盤・
  コアロジックで特に有効。task の `tests` にテスト先行か否かを明記。
- E2E/結合が要る機能は、配線 wave 後に通す結合テストを最終検証として用意する。

## 順序付けの注意
- `deps` を尊重しつつ wave を作る。循環依存が出たら設計に戻る(契約の切り方が悪い)。
- DB マイグレーション・スキーマ変更は最初の wave(基盤)に置き、以降の task はマイグレーション後の形を前提にする。
- 破壊的になりうる既存編集(共有ファイル)は配線 wave に隔離し、段階検証する。

## planning アウトプット（workflow へ渡す形）
- `waves`: task 配列の配列。各 wave は並列実行可能(配線 wave は SERIAL フラグ付き)。
- 各 task に `spec / files / deps解決済み / tests / verify` が埋まっていること。
- 受け入れ条件 → 最終検証(結合/動作確認)の対応表。
