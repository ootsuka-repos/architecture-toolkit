# design.md — 既存整合な設計の立て方（Phase 1）

新機能は「既存の設計に馴染ませる」のが最優先。本書は、既存パターンを調べ、配置・レイヤ・インターフェース・
影響範囲を確定するまでの手順をまとめる。設計のアウトプットは Phase 2(planning)が消費する。

## 1. 既存パターンの調査（最初にやる）
新規発明の前に、似た機能が既にあるかを必ず探す。あれば**その構造をテンプレートとして踏襲**する。

- **同種機能の特定**: 追加するもの(エンドポイント / 画面 / CLI コマンド / ジョブ / モデル)と同じ種類の既存実装を grep で探す。
  例: 新しい API を足すなら既存のルート定義・ハンドラ・バリデーション・テストを1つ丸ごと読む。
- **踏襲すべき要素**を抽出する:
  - ディレクトリ配置と命名規則(どの層にファイルが置かれ、どう名付けられているか)。
  - レイヤの通し方(ルート→ハンドラ→サービス→リポジトリ 等、依存方向)。
  - 入力検証・エラー処理・レスポンス整形の作法。
  - テストの置き場所・命名・スタイル(ユニット/結合/フィクスチャ)。
  - 設定・DI・ロギングの引き回し方。
- **判断**: 既存と違うやり方をするなら、その理由を1行で言えること。言えないなら既存に合わせる(一貫性 > 好み)。

## 2. 配置とレイヤ
- 新コードを**どのパッケージ/層に置くか**を決める。既存のレイヤ境界に従う。
- 依存方向(基盤→ドメイン→アプリ→UI)を守る。下位層が上位層を import しない。循環を作らない。
- 既存の公開境界(エクスポート/公開 API)に新部品をどう載せるかを決める。

## 3. 公開インターフェース設計（実装より先に契約を固める）
他 task が依存する「契約」を先に確定する。これが planning の deps の源になる。

- **シグネチャ**: 新しい関数/メソッド/型/エンドポイントの入出力を確定する(引数・戻り値・データ形)。
- **データモデル**: 新しい型/スキーマ/テーブル/DTO の形。永続化が要るならマイグレーションの要否。
- **エラー型と失敗モード**: 何が失敗しうるか、どう表現するか(例外/Result/エラーコード)。既存の失敗表現に合わせる。
- **境界の安定化**: この契約は wave0(基盤)で確定させ、依存する task はそれを所与に並列実装できるようにする。

## 4. データ/制御フローと接続点(seam)
- 入力→処理→出力の経路を1本書く(主シナリオ)。
- 既存コンポーネントとの**接続点**を特定する: どこから呼ばれ、何を呼ぶか。配線が要る箇所(routing/DI/event 登録)を列挙。

## 5. 影響範囲の洗い出し
新規ファイルだけでなく、**既存のどのファイルに手が入るか**を列挙する(これが配線 wave になる)。
- ルーティング/エントリ登録、DI/コンテナ登録、エクスポート/インデックス、設定/環境変数、
  ドキュメント/README、既存テストへの影響。
- 各々を「新規作成 or 既存編集」でタグ付けし、ファイルパスまで落とす(planning がファイル集合として使う)。

## 6. 設計レビュー観点（過剰設計を避ける）
- **YAGNI**: 今の受け入れ条件に要らない汎用化・拡張点・抽象を入れない。
- **KISS**: 既存で足りるものを再発明しない。最小の部品で受け入れ条件を満たす。
- **複数案がある場合**: トレードオフ(労力/リスク/拡張性)を併記し、**推奨を1つ**明示。判断が分かれるなら確認を取る。

## 設計アウトプット（planning へ渡す形）
- 確定した契約(型/シグネチャ/エラー型)。
- 新規作成ファイル一覧 + 既存編集ファイル一覧(パス付き、配線箇所を明示)。
- task 候補の素案(各 task が触れるファイルと依存の見当)。
- 受け入れ条件 → 検証方針(どのテスト/動作確認で満たすか)。
