# verify.md — 検証コマンドと成功条件（Phase 0 / 各 wave 後）

ベースライン(Phase 0)・各 wave 完了後・最終(受け入れ確認)で走らせる検証。**緑を確認してからコミット・次段へ**。
言語/スタックは実リポジトリに合わせて選ぶ(下は代表例)。まずプロジェクトのビルド/テスト定義
(`package.json` / `pyproject.toml` / `Makefile` / CI 設定)を見て、実際のコマンドを採用する。

## 検証レイヤ（軽い順に）
1. **ビルド/型** — 型エラー・構文エラーがないこと。
2. **リンタ/整形** — スタイル違反がないこと。
3. **テスト** — 新規テスト含め全グリーン。新機能の受け入れ条件をテストが担保していること。
4. **起動/結合スモーク** — 実際にアプリ/エンドポイント/コマンドが起動し、主シナリオが通ること。
5. **受け入れ条件チェック** — Phase 0 の Acceptance Criteria を1つずつ満たすか(可能なら実動作で)。

## 言語別の例

### TypeScript / Node
- 型: `npx tsc --noEmit`
- リンタ: `npx eslint .`
- テスト: `npm test`（または `npx vitest run` / `npx jest`）
- 起動スモーク: `npm run build && npm start`（または dev サーバを上げて主要ルートに 1 リクエスト）

### Python
- 型: `mypy .`（設定があれば）
- リンタ: `ruff check .`（または `flake8`）
- テスト: `pytest -q`
- 起動スモーク: エントリを import / CLI を `--help` 実行 / サーバを上げて 1 リクエスト

### Go
- ビルド: `go build ./...`
- テスト: `go test ./...`
- 静的: `go vet ./...`

### Rust
- ビルド: `cargo build`
- テスト: `cargo test`
- 静的: `cargo clippy`

## 成功条件
- 上記の採用したレイヤがすべて 0 エラーで完了する。
- 新機能のテストが存在し、緑である(テスト無しの「動いたはず」は不可)。
- ベースラインで通っていた既存テストを壊していない(回帰なし)。
- 受け入れ条件が満たされている(実動作 or テストで確認)。

## 失敗時
- **そこで停止**(次 wave に進まない)。当該 wave の変更を切り分け、原因 task を特定する。
- subagent の `verify_output` と main 側の検証ログを突き合わせ、`failed` の task を報告する。
- 安易に進めない。緑に戻してからコミット・次段へ。
